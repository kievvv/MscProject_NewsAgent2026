# 产品重构与用户故事草案

## 1. 文档目的

这份文档用于固化当前项目下一阶段的产品思考，帮助我们从“功能集合”转向“任务驱动的 AI 金融研究与交易辅助平台”。

目标有三类：

- 明确产品定位与差异化
- 将现有功能重组为清晰的用户工作流
- 构造一个适合 presentation 的代表性用户故事


## 2. 当前项目的本质与特殊性

### 2.1 当前项目已经具备的核心能力

从后端能力出发，当前项目已经不是单纯的新闻网站，而是具备了以下能力：

- 实时接入 Telegram 高质量 Crypto 新闻源
- 历史回填 + 实时监听 + 队列 + 入库
- 关键词提取、趋势分析、相似度分析、摘要生成
- 市场数据整合
- AI Agent 对话
- AI Skill 工作流
- 搜索、筛选、推荐、用户画像雏形

### 2.2 项目的真正特殊之处

这个项目最有价值的点，不是“有新闻”，也不是“有聊天框”，而是：

- 新闻数据是实时接入的，而不是静态文章集合
- 数据进入系统后可以立刻被小模型分析
- 大模型 Agent 可以把这些能力统一编排为面向用户的工作流

因此，本产品不应被定义为“资讯平台”，而应定义为：

**一个以 Agent 为核心交互入口、以实时新闻与多模型分析为底座的 AI 金融研究与交易辅助平台。**


## 3. 当前产品存在的关键问题

### 3.1 功能存在，但没有形成工作流

目前的问题不是功能太少，而是：

- 用户不知道进来之后第一步做什么
- Agent、Skill、搜索、图表、新闻列表之间没有统一逻辑
- 用户必须自己理解系统结构，主动拼接功能

这会导致产品看起来像 demo：

- 页面很多
- 模块很多
- 但用户价值链条不清晰

### 3.2 前端仍然是“资讯堆叠逻辑”

目前前端更像：

- 一堆图表
- 一堆新闻
- 一堆参数
- 一个额外的 AI 入口

而不是：

- 一个 AI 原生的任务入口
- 一个帮用户节省时间、形成判断、指导下一步动作的工作台

### 3.3 Agent 与 Skill 没有成为主流程

目前 Agent 和 Skill 的问题不是没有实现，而是没有融入使用路径：

- Agent 只是聊天框
- Skill 只是功能按钮

理想状态应该是：

- 用户只表达需求
- Agent 理解任务
- Skill 在后台自动被调用
- 最终以报告、推荐、筛选结果、下一步建议的形式呈现


## 4. 下一阶段的产品核心理念

### 4.1 总体理念

未来产品形态应遵循：

**大模型 Agent 负责统筹交互与任务编排，小模型与传统 NLP 模型负责稳定的数据处理与分析执行。**

也就是：

- 大模型负责理解用户意图、组织工作流、生成解释、做个性化交互
- 小模型负责关键词、摘要、趋势、实体、相似度、规则提取等高频低成本任务

### 4.2 交互理念

用户的核心交互不应该是：

- 手动点筛选器
- 自己选参数
- 自己搜索新闻
- 自己理解图表

而应该是：

- “推荐一下适合我的新闻”
- “分析一下今天市场局势”
- “我只想看过去 6 小时值得短线关注的变化”
- “帮我筛出今天与 BTC 和 ETF 最相关、影响最大的新闻”

由 Agent 把这些自然语言转成：

- 结构化筛选条件
- 自动调用搜索与分析服务
- 自动选择合适的 Skill
- 最终返回报告、列表和结论


## 5. 产品重构方向

### 5.1 产品定位

建议将产品定位为：

**AI 驱动的实时金融情报与交易研究工作台**

它不是传统新闻平台，也不是纯量化平台，而是连接以下三层：

- 实时信息获取
- 智能情报处理
- 研究与交易决策支持

### 5.2 核心价值主张

给用户带来的直接价值应该是：

- 帮用户节省看新闻和筛新闻的时间
- 帮用户理解新闻背后的市场意义
- 帮用户根据自己的风格看到更适合的信息
- 帮用户从“知道发生了什么”走向“理解该怎么判断”


## 6. 建议采用的产品结构

### 6.1 三层结构

#### 第一层：AI 任务入口层

这一层是用户最先接触的核心区域。

主要功能：

- 自然语言输入
- 常用任务快捷入口
- 个性化推荐入口
- 今日重点任务入口

典型入口文案：

- 推荐适合我的新闻
- 分析今天市场局势
- 我想看高风险高收益机会
- 帮我做一份 BTC 相关快报
- 过去 4 小时哪些消息最值得关注

#### 第二层：情报与研究层

这一层承接 Agent 调度后的结果。

主要功能：

- 新闻聚合结果
- 热点主题
- 事件时间线
- 关键词趋势
- 主题深挖报告
- 风险/机会摘要

#### 第三层：策略与行动层

这一层是产品未来向交易辅助深入的部分。

主要功能：

- 信号观察
- 自定义 watchlist
- 策略研究
- 回测与模拟交易
- 多策略对比


## 7. Agent 与 Skill 的重构原则

### 7.1 Agent 的角色

Agent 不再只是聊天机器人，而是整个平台的操作系统层。

Agent 要负责：

- 理解用户任务
- 识别当前上下文
- 调用相关搜索、分析、推荐、报告、策略模块
- 将结果组织成用户能直接消费的输出

### 7.2 Skill 的角色

Skill 不应作为“独立按钮功能”存在，而应成为 Agent 的后台执行单元。

未来用户使用方式应该是：

- 用户提出任务
- Agent 识别任务类型
- 自动决定是否调用 `DailyBriefing`、`DeepDive`、`AlphaHunter`
- 输出统一格式的研究结果

也就是说：

- Agent 是前台交互者
- Skill 是后台流程引擎

### 7.3 典型 Skill 重构方向

建议未来保留并强化以下类型：

- Personalized Briefing
- Market Pulse
- Theme Deep Dive
- Opportunity Radar
- Risk Watch
- Strategy Draft Lab


## 8. 用户工作流设计

### 8.1 核心工作流

建议所有前端呈现最终围绕一个主工作流：

**发现 -> 理解 -> 过滤 -> 深挖 -> 形成判断 -> 行动验证**

#### 阶段一：发现

系统自动告诉用户：

- 今天发生了什么
- 哪些是与你相关的
- 哪些是需要优先关注的

#### 阶段二：理解

系统帮助用户解释：

- 为什么重要
- 影响哪些资产/主题
- 市场情绪如何
- 当前是风险还是机会

#### 阶段三：过滤

系统根据用户画像与自然语言意图筛选：

- 适合短线的信息
- 适合稳健配置的信息
- 适合高风险偏好的热点

#### 阶段四：深挖

如果用户继续追问，进入：

- 主题报告
- 相关新闻证据链
- 趋势分析
- 市场因子整合

#### 阶段五：形成判断

系统给出：

- 关键结论
- 支撑证据
- 风险提示
- 后续建议动作

#### 阶段六：行动验证

未来进一步进入：

- 自定义观察
- 策略草稿
- 回测与模拟盘


## 9. 个性化与记忆系统设计

### 9.1 为什么必须做记忆系统

因为平台的目标不是“所有人都看到同样的首页”，而是：

- 不同用户看到不同的推荐
- 不同用户收到不同风格的报告
- Agent 能记住用户偏好与上下文

### 9.2 建议采用长短期记忆结合

#### 短期记忆

短期记忆服务于当前会话。

可存储：

- 当前对话主题
- 最近关注资产
- 最近查询条件
- 当前任务上下文

用途：

- 保持多轮交互连续性
- 在一次任务里不重复问用户

#### 长期记忆

长期记忆服务于个性化。

可存储：

- 风险偏好
- 投资周期
- 关注资产/主题
- 交易频率
- 偏好的输出形式
- 过去行为统计

用途：

- 个性化推荐
- 个性化报告
- 个性化首页
- 个性化推送

### 9.3 用户画像建议字段

第一版不需要太复杂，可以先有这些字段：

- user_type
- risk_preference
- trading_frequency
- holding_period
- focus_assets
- focus_themes
- report_style
- signal_sensitivity

### 9.4 用户画像分层建议

可以先做四类典型用户：

- 高频激进型
- 中短线机会型
- 中长期研究型
- 稳健保守型


## 10. 未来后端功能扩展方向

### 10.1 近期优先

- 自然语言转筛选模板
- 个性化推荐引擎
- 报告模板化生成
- 事件聚合与主题聚类
- 新闻重要性评分
- 推送机制

### 10.2 中期增强

- 链上数据接入
- 更多交易所公告与公开信息
- 更多金融分析因子
- 市场信号评分与置信度机制
- 更强的报告 Agent

### 10.3 后期扩展

- 自然语言生成量化策略
- `freqtrade` 接入
- 多策略回测与对比
- 模拟交易与策略解释

这一部分建议独立为：

**AI Strategy Lab**

而不要一开始就把它混入主页面主流程。


## 11. Presentation 用的代表性用户故事

### 11.1 为什么用户故事重要

最终 presentation 需要在几分钟内清楚说明：

- 用户是谁
- 他有什么痛点
- 为什么需要我们的系统
- 系统如何从头到尾帮助他完成任务

因此需要一个既真实、又有代表性、又能串起全部功能的用户故事。

### 11.2 建议的典型用户画像

建议使用以下人物作为 presentation 主角：

#### 用户名称

Leo

#### 身份设定

- 28 岁
- 个人加密交易者
- 同时兼顾研究和交易
- 每天都会看市场，但没有时间刷完所有新闻

#### 投资风格

- 风险偏好中高
- 关注热点主题和中短线机会
- 会看 BTC/ETH，也关注 AI、模块化、基础设施等叙事
- 希望尽快知道“哪些新闻值得看，为什么值得看”

#### 典型痛点

- 信息太多，渠道太分散
- 同样的新闻看很多遍，浪费时间
- 很难快速判断一条新闻是否值得行动
- 缺少一个能持续理解自己偏好的研究助手

### 11.3 为什么这个用户最有代表性

这个画像适合当前产品，因为它同时需要：

- 实时新闻
- 智能筛选
- 个性化推荐
- 报告生成
- AI 交互
- 后续策略研究

也就是说，他既能串起“新闻情报”主线，也能自然过渡到“策略实验室”愿景。


## 12. 建议的 Presentation 用户故事脚本

### 12.1 故事主线

Leo 早上打开平台，并不是去刷新闻，而是直接进入 AI 工作台。

他对 Agent 说：

“我今天主要关注 BTC、AI 和高波动机会，帮我快速看一下过去 12 小时最重要的市场变化。”

系统完成以下流程：

1. Agent 理解 Leo 的需求
2. 结合 Leo 的长期画像和当前任务上下文
3. 从实时新闻流中自动筛出最相关的新闻
4. 用小模型做关键词、摘要、主题聚类、趋势判断
5. 用大模型整合成一份个性化市场简报
6. 告诉 Leo：
   - 今天最重要的 3 个主题
   - 每个主题为什么重要
   - 可能影响哪些币种和市场方向
   - 哪些适合短线关注，哪些适合继续观察

### 12.2 第二步交互

Leo 接着问：

“第二条主题帮我深挖一下，重点看是否有交易机会。”

系统：

- 自动触发 Deep Dive 类 Skill
- 展示相关新闻证据链
- 展示关键词趋势
- 结合市场数据和风险提示
- 输出一份深挖报告

### 12.3 第三步交互

Leo 进一步说：

“如果我想围绕这个主题做一个短线策略，可以怎么验证？”

未来产品中，系统将：

- 把这个研究主题转成策略草稿
- 用自然语言生成 `freqtrade` 策略模板
- 提供模拟回测结果
- 对比不同策略版本

这样，用户就完成了从：

**新闻获取 -> 信息理解 -> 主题研究 -> 交易验证**

的一条完整链路。

### 12.4 Presentation 中强调的价值

在 presentation 中，这个故事可以非常清晰地体现：

- 平台不是在堆新闻
- 平台是在节省用户的认知成本
- 平台通过 Agent 串起所有功能
- 平台能根据用户风格进行个性化服务
- 平台未来能够延伸到策略验证与模拟交易


## 13. 产品重构建议结论

### 13.1 结论一

接下来的重构方向不应再以“补页面”为中心，而应以：

**重构用户任务流**

为中心。

### 13.2 结论二

大模型 Agent 应成为整个平台的主入口和统筹层，小模型承担执行分析与结构化处理。

### 13.3 结论三

前端不应首先展示参数和图表，而应首先展示：

- AI 引导入口
- 个性化推荐
- 今日重点任务
- 为用户筛好的结果

### 13.4 结论四

AI Skill 不应继续孤立存在，而应被 Agent 编排进用户主流程。

### 13.5 结论五

为了 presentation，最重要的不是展示所有功能，而是通过一个高代表性的用户故事，把系统能力串成一条清晰、有说服力的工作流。


## 14. 下一步建议

建议我们下一轮讨论聚焦以下三个问题：

1. 首页应该承接哪个任务流作为主入口
2. 用户画像第一版字段和推荐逻辑如何定义
3. Presentation 的 3-5 分钟演示流程如何逐页展开
